## 核心设计：data 层架构

### 设计模式

`Collector(Fetcher(DataType))` — 两层依赖注入 + 组合

| 层 | 角色 | 模式 |
|----|------|------|
| `Collector` | 编排采集，只依赖 `FetcherProtocol` | 策略模式 |
| `AkShareFetcher(DailyKline)` | 封装数据源，内部按 data_type 分发 | 工厂分发 |
| `BaseData` | 所有采集数据的基类 | 多态基类 |
| `FetcherProtocol` | 采集行为的接口约束 | Protocol（结构化子类型） |

### 数据流

```
AkShare API
  → AkShareFetcher._fetch_daily_kline()
  → KlineParser.parse_kline_df()       # DataFrame(中文列) → list[DailyKline]
  → Collector.collect()                # 返回 list[BaseData]
  → StockService.collect_and_save()    # 去重写入 daily_klines 表
```

### 扩展规则

- **新增数据源（如 tushare）**：新建 `data/adapters/tushare/fetcher.py`，实现 `fetch(params)` 方法即满足 `FetcherProtocol`，无需修改已有代码
- **新增数据类型（如新闻）**：新建 `data/schemas/news.py`（继承 `BaseData`），在对应 Fetcher 里加 `_fetch_news` handler 并注册到 `_dispatch`

## app 层关键约定

### 数据库

- ORM 基类：`app/database/base.py` → `Base = declarative_base()`
- Session 管理：`app/database/connection.py` → `get_db_session()`（`@contextmanager`，同时兼容 FastAPI `Depends` 和普通 `with` 语法）
- 启动建表：`main.py` 的 `lifespan` 里调用 `Base.metadata.create_all(bind=engine)`

### 命名约定

| 数据概念 | data 层 | app/database 层 | app/schemas 层（API） |
|---------|---------|-----------------|----------------------|
| 日K线 | `DailyKline(BaseData)` | `DailyKlineDB(Base)` | `DailyKlineResponse` |
| 表名 | — | `daily_klines` | — |

### 两个 StockService 说明

`data/services/stock_service.py` 是**唯一的** StockService，`app/services/` 目录当前为空（已删除无用的中间层）。API 路由直接 import `data.services.stock_service.StockService`。
